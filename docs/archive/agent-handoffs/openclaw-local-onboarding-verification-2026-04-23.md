# OpenClaw Local Onboarding Verification

**Date:** 2026-04-23
**Bead:** `omniweb-agents-r9fn`
**Scope:** Reproduce the real local/operator onboarding path for the shipped OpenClaw research-agent bundle using the local OpenClaw CLI, record the exact command flow, and capture the blockers that must shape repo-facing onboarding docs.

## 1. Findings First

1. The cold-start CLI entrypoint is `openclaw onboard`, not `openclaw setup`, when running non-interactively. `setup --non-interactive` fails fast and explicitly tells the operator to rerun `onboard --accept-risk`.
2. A shipped OmniWeb bundle can be wired as an OpenClaw workspace without copying files elsewhere. The research-agent bundle at `packages/omniweb-toolkit/agents/openclaw/research-agent/` is a real workspace surface.
3. The most reliable local-skill verification command is `openclaw skills info omniweb-research-agent`, not the broad `skills list` table.
4. `openai-codex` is currently **interactive-only** in OpenClaw onboarding. A fully non-interactive local path cannot rely on that auth choice.
5. A non-interactive agent can still be created against the OmniWeb bundle with `openclaw agents add ... --workspace ...`.
6. The first local turn currently fails on model auth, not on workspace loading or skill discovery. The observed blocker is missing provider auth for `openai/gpt-5.4`.
7. OpenClaw mutates the workspace bundle on first agent wiring by materializing files like `AGENTS.md`, `TOOLS.md`, `SOUL.md`, `USER.md`, `HEARTBEAT.md`, and `.openclaw/workspace-state.json`. Those files are generated by the CLI and should not be treated as committed source-of-truth unless intentionally exported later.

## 3. Workspace Mutation Observed On First Wiring

OpenClaw wrote these files into the research-agent bundle workspace during onboarding/agent creation:

- `AGENTS.md`
- `TOOLS.md`
- `SOUL.md`
- `USER.md`
- `HEARTBEAT.md`
- `.openclaw/workspace-state.json`

These are real runtime outputs from the OpenClaw side, not committed package artifacts in `main` at the time this verification run started.

Implications:

1. onboarding docs should warn operators that OpenClaw may materialize workspace-local memory/bootstrap files on first run
2. generated workspace files should not be committed accidentally from a local dogfood session
3. if the repo later chooses to pre-export some of these surfaces, that should happen deliberately via the package export pipeline, not by copying accidental local run output

## 4. What This Forces In Repo Docs

### 4.1 Root onboarding docs should say

- for a first local install from GitHub clone, use the shipped bundle under `packages/omniweb-toolkit/agents/openclaw/<archetype>/`
- cold-start onboarding is:
  1. clone repo
  2. `npm install`
  3. run `openclaw onboard` or `openclaw setup` depending on whether you are doing full first-time config or just pointing an already-configured profile at a workspace
  4. add or point an isolated agent at the bundle workspace
  5. verify with `openclaw skills info <slug>`
  6. only then try `openclaw agent --local`

### 4.2 Root onboarding docs should not say

- "point OpenClaw at GitHub" as if GitHub URL alone is a normal install path
- "verify with `openclaw skills list`" as the primary check
- "non-interactive `openai-codex` setup works today"
- "a clone alone is sufficient for a successful local turn"

### 4.3 Current honest blocker statement

The repo is **OpenClaw-workspace consumable** today, but a successful local turn still depends on a model-auth path being configured for the isolated agent. On this machine:

- `openai-codex` onboarding is interactive-only
- a new isolated agent defaults to `openai/gpt-5.4` unless explicitly configured otherwise
- that default path fails without a provider API key in the agent auth store

## 5. Recommended Follow-Up Work

1. Add a root `OPENCLAW.md` or equivalent root README section that documents the verified local-bundle path.
2. Change onboarding verification language from `skills list` to `skills info <slug>`.
3. Make the model/auth split explicit:
   - workspace/skill wiring
   - then provider auth
   - then local turn
4. Keep package bundle README cleanup separate until the overlapping `#269` / `#270` surface lands or is superseded.
5. Treat a future successful local turn as a separate proof artifact, not as something inferred from workspace discovery alone.
